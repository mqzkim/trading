In `get_signal()` (around lines 73-78), the numeric branch sets `strength_val = float(strength_raw)` without
normalizing. When the handler returns a 0-100 float (e.g. 25.0), this violates `SignalResponse.strength: float = Field(ge=0, le=1)`.

Fix: replace the numeric branch so it normalizes to [0, 1]:
```python
strength_raw = data.get("strength", "MODERATE")
if isinstance(strength_raw, (int, float)):
    strength_val = min(1.0, max(0.0, float(strength_raw) / 100.0))
else:
    strength_map = {"STRONG": 0.9, "MODERATE": 0.5, "WEAK": 0.2}
    strength_val = strength_map.get(str(strength_raw).upper(), 0.5)
```

Only modify the `strength_raw` assignment block. Do NOT touch direction mapping, methodology_votes construction, or any other part of the router.

Root cause (from UAT): `SignalStrength.value` lives in the domain as a 0-100 scale. The handler returns it as-is. The string-to-float map (STRONG/MODERATE/WEAK) was already correct for string inputs. The numeric path just needed `/ 100.0`.

Also add the following test to `tests/unit/test_api_v1_signals.py` to exercise the numeric strength path (the actual bug being fixed):

```python
def test_numeric_strength_divided_by_100(client, auth_headers):

    assert response.status_code == 200
    body = response.json()

```
