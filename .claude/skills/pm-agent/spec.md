## 1. 개요

### 1.1 배경

1인 개발자가 7개 제품을 동시에 운영하려면, "무엇을 만들 것인가"에서 "어떤 순서로, 어떤 규모로, 어떤 기술로"까지 구체화하는 과정이 필수다. idea.md 같은 전략 문서에는 비전과 방향성이 담겨 있지만, 이를 **실행 가능한 프로젝트 단위**로 분해하는 작업은 경험 많은 PM(Product Manager)의 역할이다.

현재 에이전트 시스템에는 이 간극이 존재한다:

```
idea.md (전략/비전)
       │
       ▼
  ??? (PM 역할 부재) ← 이 에이전트가 메운다
       │
       ▼
scaffolding (프로젝트 생성)
```

Portfolio Manager(0-1)는 **운영 중인** 제품의 성과를 분석하고 자원 배분을 추천한다.
PM Agent(0-4)는 **아직 만들어지지 않은** 아이디어를 프로젝트로 구체화하는 역할이다.

```
PM Agent (0-4)               Portfolio Manager (0-1)
──────────────               ─────────────────────
아이디어 → 프로젝트 스펙       운영 중인 제품 → 성과 분석
"무엇을 만들까?"              "어디에 집중할까?"
프리-프로덕션                  포스트-런칭
```

### 1.2 목표

| 목표 | 설명 |
|------|------|
| **아이디어 구조화** | 비정형 아이디어 문서에서 핵심 가치 제안, 타겟 고객, 수익 모델을 추출 |
| **프로젝트 분해** | 하나의 아이디어를 독립 배포 가능한 프로젝트 단위로 분해 |
| **실행 스펙 생성** | 각 프로젝트에 대해 기술 요구사항, 마일스톤, 성공 지표를 포함한 스펙 생성 |
| **우선순위 결정** | SWIFT 프레임워크 + 수익/노력 매트릭스로 실행 순서 추천 |
| **에이전트 매핑** | 각 프로젝트가 필요로 하는 에이전트 조합을 자동 매핑 |
| **포트폴리오 시각화** | 전체 프로젝트 포트폴리오를 한눈에 조감할 수 있는 뷰 제공 |

### 1.3 비-목표 (Non-Goals)

- 코드 생성 (scaffolding 에이전트의 역할)
- 이미 운영 중인 제품의 성과 분석 (portfolio-manager의 역할)
- 시장 조사 자동화 (수동 입력 또는 web-scraper 연동)
- 투자 유치/사업 계획서 작성 (B2B 컨설팅 영역)

## 2. 설계 원칙

### 2.1 PM의 핵심 역량을 인코딩

```
30년 PM/아키텍트 경험에서 나온 핵심 판단 기준:

1. [실현 가능성] 2주 안에 MVP를 출시할 수 있는가?
2. [시장 검증]   돈을 낼 사람이 구체적으로 누구인가?
3. [기술 적합성] 현재 기술 스택으로 만들 수 있는가?
4. [독립성]      다른 프로젝트의 성공에 의존하지 않는가?
5. [배포 채널]   첫 100명의 사용자를 어디서 데려올 수 있는가?
6. [비용 구조]   수익 없이도 운영 비용이 감당 가능한가?
```

### 2.2 "가장 작은 유의미한 단위"로 분해

아이디어를 분해할 때의 핵심 원칙:

```
BAD:  "IDEA 2: API 서비스" → 하나의 거대 프로젝트
GOOD: "IDEA 2: API 서비스" → 3개의 독립 프로젝트
      ├── Project A: AI 스크래핑 API (Week 1-2)
      ├── Project B: 문서 파싱 API (Week 3-4)
      └── Project C: 웹→마크다운 API (Week 5-6)
```

분해 기준:
- 각 프로젝트는 **독립적으로 출시 가능**해야 한다
- 각 프로젝트는 **독립적으로 수익을 창출**할 수 있어야 한다
- 각 프로젝트의 MVP는 **2주 이내**에 완성 가능해야 한다
- 프로젝트 간 공유 자원(DB, 인증, 인프라)은 명시적으로 표시한다

### 2.3 Daniel Vassallo의 "Small Bets" 원칙 적용

```
┌─────────────────────────────────────────────────────┐
│  PM Agent의 핵심 마인드셋                              │
│                                                      │
│  "어떤 하나의 프로젝트가 실패해도 죽지 않는 구조"        │
│                                                      │
│  • 각 프로젝트의 최대 투자 시간: 2주                    │
│  • 각 프로젝트의 최대 초기 비용: $50/월                 │
│  • 3개월 내 의미 있는 시그널 없으면 → 유지보수 모드     │
│  • 포트폴리오 전체에서 1~2개만 히트해도 성공             │
└─────────────────────────────────────────────────────┘
```

## 3. Skill 구조

### 3.1 디렉토리 레이아웃

```
.claude/skills/pm-agent/
├── SKILL.md                          # 메인 진입점 (~250줄)
├── spec.md                           # 상세 스펙 (이 문서)
│
├── frameworks/                       # 분석 프레임워크 (필요시 로드)
│   ├── swift-validation.md           # SWIFT 프레임워크 상세 가이드
│   ├── revenue-model-patterns.md     # 수익 모델 패턴 카탈로그
│   ├── tech-stack-decision.md        # 기술 스택 의사결정 트리
│   └── distribution-channels.md      # 배포 채널 전략 가이드
│
├── templates/                        # 출력 템플릿
│   ├── project-spec-template.md      # 프로젝트 스펙 템플릿
│   ├── portfolio-overview-template.md # 포트폴리오 조감도 템플릿
│   └── milestone-template.md         # 마일스톤 계획 템플릿
│
└── examples/                         # 참조 예시
    └── idea-md-decomposition.md      # idea.md 분해 예시
```

### 3.2 점진적 공개 전략

```
[항상 로드] SKILL.md (name + description + 핵심 워크플로우)
     │
     ├─ [1단계] SKILL.md 본문 (~250줄)
     │   핵심 분석 프로세스, 출력 형식, 에이전트 매핑
     │
     ├─ [2단계] frameworks/ 파일
     │   분석 깊이에 따라 선택적 로드
     │   예: 수익 모델 상세 분석 시 → revenue-model-patterns.md
     │
     └─ [3단계] templates/ 파일
         출력물 생성 시 템플릿 참조
```

## 4. 핵심 워크플로우

### 4.1 전체 파이프라인

```
┌──────────────────────────────────────────────────────────────┐
│                    PM AGENT WORKFLOW                           │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Phase 1: 아이디어 수집 (Intake)                               │
│  ├─ 입력 문서 읽기 (idea.md, 자유 텍스트, URL 등)              │
│  ├─ 핵심 아이디어 추출 및 구조화                                │
│  └─ 기존 프로젝트/계획과 중복 확인                              │
│           │                                                   │
│  Phase 2: 아이디어 분석 (Analysis)                             │
│  ├─ 각 아이디어의 핵심 가치 제안(Value Proposition) 정의        │
│  ├─ 타겟 고객 페르소나 구체화                                   │
│  ├─ 경쟁 환경 분석 (있는 정보 기반)                             │
│  ├─ SWIFT 프레임워크 검증                                      │
│  └─ 수익 모델 패턴 매칭                                        │
│           │                                                   │
│  Phase 3: 프로젝트 분해 (Decomposition)                        │
│  ├─ 아이디어 → 독립 프로젝트 단위로 분해                        │
│  ├─ 프로젝트 간 의존성 맵 생성                                  │
│  ├─ 공유 자원(인프라, 컴포넌트) 식별                            │
│  └─ MVP 스코프 정의 (각 프로젝트별)                             │
│           │                                                   │
│  Phase 4: 스펙 생성 (Specification)                            │
│  ├─ 프로젝트별 상세 스펙 문서 생성                              │
│  ├─ 기술 요구사항 + 아키텍처 결정 사항                          │
│  ├─ 마일스톤 + 타임라인                                        │
│  ├─ 성공 지표(KPI) 정의                                        │
│  └─ 리스크 분석 + 킬 크라이터리어                               │
│           │                                                   │
│  Phase 5: 우선순위 & 로드맵 (Prioritization)                   │
│  ├─ ICE 스코어 (Impact × Confidence × Ease) 산출              │
│  ├─ 포트폴리오 레이어 배치 (Layer 1/2/3)                       │
│  ├─ 실행 순서 및 병렬화 가능 여부                               │
│  └─ 12개월 로드맵 생성                                         │
│           │                                                   │
│  Phase 6: 에이전트 매핑 & 핸드오프 (Handoff)                   │
│  ├─ 각 프로젝트 × 필요 에이전트 매트릭스                        │
│  ├─ scaffolding 에이전트 호출 파라미터 생성                     │
│  └─ 실행 계획 문서를 plans/ 디렉토리에 저장                     │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 4.2 Phase별 상세

#### Phase 1: 아이디어 수집

입력 소스 처리 규칙:

| 입력 유형 | 처리 방법 |
|-----------|-----------|
| 파일 경로 (`.md`, `.txt`) | Read 도구로 전체 내용 읽기 |
| URL | WebFetch로 콘텐츠 추출 |
| 인라인 텍스트 | 직접 파싱 |
| 복합 (파일 + 추가 지시) | 파일 내용 + 사용자 지시 병합 |

중복 확인 프로세스:
1. `plans/` 디렉토리에서 관련 기존 계획 검색
2. `.claude/skills/` 에서 유사 기능 에이전트 확인
3. 중복 발견 시 사용자에게 알림 → 병합/신규 선택

#### Phase 2: 아이디어 분석

각 아이디어에 대해 **5W1H + SWIFT** 분석 수행:

```
WHO   — 누가 사용하는가? (구체적 페르소나)
WHAT  — 무엇을 해결하는가? (핵심 페인 포인트)
WHY   — 왜 지금인가? (시장 타이밍)
WHERE — 어디서 고객을 찾는가? (배포 채널)
WHEN  — 언제까지 MVP를 출시하는가? (타임라인)
HOW   — 어떻게 돈을 버는가? (수익 모델)
```

#### Phase 3: 프로젝트 분해

분해 알고리즘:

```
1. 아이디어에서 독립 가치 단위(Independent Value Unit) 추출
   └─ "이것만으로도 사용자가 돈을 낼 수 있는가?"

2. 각 IVU를 프로젝트로 승격
   └─ 프로젝트 이름, 코드명, 한줄 설명 부여

3. 프로젝트 간 의존성 분석
   ├─ 기술적 의존성 (공유 DB, 공유 인증, 공유 API)
   ├─ 비즈니스 의존성 (A의 사용자가 B의 잠재 고객)
   └─ 시간적 의존성 (A를 먼저 해야 B가 가능)

4. 의존성 그래프에서 병렬화 가능 프로젝트 식별
   └─ 동시에 진행 가능한 프로젝트 그룹 생성

5. 각 프로젝트의 MVP 스코프 정의
   └─ "2주 안에 출시할 수 있는 최소 기능 세트"
```

#### Phase 4: 스펙 생성

각 프로젝트별 생성 항목:

| 섹션 | 내용 |
|------|------|
| **프로젝트 개요** | 이름, 한줄 설명, 핵심 가치 제안, 타겟 고객 |
| **문제 정의** | 해결하려는 문제, 현재 대안, 차별화 포인트 |
| **기술 요구사항** | 기술 스택, DB 스키마 개요, API 엔드포인트 목록, 외부 의존성 |
| **MVP 기능 목록** | Must-have / Nice-to-have / Out-of-scope 분류 |
| **수익 모델** | 가격 책정, 무료 티어, 유료 플랜, 예상 전환율 |
| **마일스톤** | Week 1 / Week 2 / Month 1 / Month 3 구체적 목표 |
| **성공 지표 (KPI)** | 1개월 / 3개월 / 6개월 목표 수치 |
| **킬 크라이터리어** | "이 조건이면 포기한다" 명확한 기준 |
| **리스크** | 기술적/시장적/운영적 리스크와 완화 전략 |
| **에이전트 매핑** | 필요한 에이전트 목록과 호출 시점 |

#### Phase 5: 우선순위 결정

**ICE 스코어링:**

```
Impact (영향력)     1~10  — 성공 시 예상 월 수익 기준
Confidence (확신도) 1~10  — 시장 검증 수준 + 기술 실현 가능성
Ease (용이성)       1~10  — MVP 소요 시간 + 비용

ICE Score = Impact × Confidence × Ease / 100
```

**포트폴리오 레이어 배치:**

```
Layer 1 (즉시 시작):    ICE ≥ 6.0 AND Ease ≥ 7
Layer 2 (1~3개월 후):   ICE ≥ 4.0 AND Impact ≥ 6
Layer 3 (3~6개월 후):   나머지 (장기 투자)
```

#### Phase 6: 에이전트 매핑 & 핸드오프

각 프로젝트에 필요한 에이전트를 라이프사이클별로 매핑:

```
프로젝트 라이프사이클    필요 에이전트
─────────────────    ──────────────
기획 단계              pm-agent (자기 자신)
개발 단계              scaffolding → code-review → test-generator → doc-generator
데이터 수집            web-scraper → data-transformer → data-enricher
콘텐츠/마케팅          content-writer → seo-optimizer → social-media → product-launch
배포/운영              deployment → health-monitor → alert → cost-tracker
고객/비즈니스          customer-support → onboarding → revenue-analytics → churn-prevention
```

## 5. 아이디어 분석 프레임워크

### 5.1 SWIFT 검증 상세

idea.md에 정의된 SWIFT 프레임워크를 정량적으로 평가:

| 기준 | 점수 기준 | 1점 | 5점 | 10점 |
|------|-----------|-----|-----|------|
| **S**elf-serve | 고객이 영업 없이 구매 가능 | 영업팀 필요 | 일부 셀프서브 | 100% 셀프서브 |

## 6. 프로젝트 분해 엔진

### 6.1 분해 규칙

**규칙 1: 독립 배포 가능성 (Independent Deployability)**

```
GOOD: 각 프로젝트가 자체 도메인 + 자체 DB로 운영 가능
BAD:  프로젝트 B가 프로젝트 A의 DB에 의존
```

**규칙 2: 독립 수익 가능성 (Independent Revenue)**

```
GOOD: 스크래핑 API만으로도 유료 사용자 확보 가능
BAD:  디렉토리가 없으면 API가 무용지물
```

**규칙 3: 2주 MVP 규칙 (Two-Week MVP)**

```
GOOD: 핵심 기능 3개로 출시 가능
BAD:  기본 기능만 해도 4주 이상 소요 → 더 작게 나눠야 함
```

**규칙 4: 공유 자원 명시 (Shared Resource Identification)**

프로젝트 간 공유하는 자원을 명시적으로 추출:

```
공유 가능 자원:
├── 인증 시스템 (Clerk — 모든 프로젝트 공유 가능)
├── 결제 인프라 (Stripe — 공유 Stripe 계정)
├── UI 컴포넌트 (shadcn/ui — 공유 디자인 시스템)
├── 기본 보일러플레이트 (IDEA 1의 출력물)
└── 도메인/DNS (Vercel — 서브도메인 활용)
```

### 6.2 분해 패턴

**패턴 A: 수직 분해 (Vertical Slicing)**

하나의 아이디어를 기능별로 분리:

```
IDEA 2: API 서비스
├── Project: scrape-api     (스크래핑 API)
├── Project: doc-parse-api  (문서 파싱 API)
└── Project: web-summary-api (웹 요약 API)
```

**패턴 B: 수평 분해 (Horizontal Slicing)**

하나의 아이디어를 고객 세그먼트별로 분리:

```
IDEA 3: 니치 디렉토리
├── Project: mcp-directory    (AI 개발자 타겟)
├── Project: ai-tools-dir     (비즈니스 의사결정자 타겟)
└── Project: kr-saas-dir      (한국 기업 타겟)
```

**패턴 C: 단계적 분해 (Phased Slicing)**

하나의 아이디어를 성숙도별로 분리:

```
IDEA 5: 바이브코딩 대행
├── Phase 1 Project: code-review-service  (비동기 리뷰 — 즉시)
├── Phase 2 Project: live-session-service (라이브 코칭 — 1개월 후)
└── Phase 3 Project: production-package   (프로덕션 패키지 — 3개월 후)
```

## 8. 우선순위 매트릭스

### 8.1 ICE 스코어링 상세

**Impact (영향력) 산출:**

| 점수 | 기준 |
|------|------|
| 9~10 | 월 $5,000+ 수익 잠재력 |
| 7~8 | 월 $2,000~$5,000 수익 잠재력 |
| 5~6 | 월 $500~$2,000 수익 잠재력 |
| 3~4 | 월 $100~$500 수익 잠재력 |
| 1~2 | 수익보다 학습/네트워크 가치 |

**Confidence (확신도) 산출:**

| 점수 | 기준 |
|------|------|
| 9~10 | 유사 제품이 이미 수익 창출 중 (벤치마크 존재) |
| 7~8 | 시장 수요 신호 존재 (검색량, 커뮤니티 요청) |
| 5~6 | 본인의 직관 + 간접 증거 |
| 3~4 | 가설 수준 |
| 1~2 | 완전한 실험 |

**Ease (용이성) 산출:**

| 점수 | 기준 |
|------|------|
| 9~10 | 1주 이내 MVP, 비용 $0 |
| 7~8 | 2주 이내 MVP, 비용 $0~$20/월 |
| 5~6 | 4주 이내 MVP, 비용 $20~$50/월 |
| 3~4 | 6주 이내 MVP, 비용 $50~$100/월 |
| 1~2 | 8주+ MVP, 비용 $100+/월 |

### 8.2 포트폴리오 배치 알고리즘

```
Input: 프로젝트 목록 + ICE 스코어
Output: Layer 1/2/3 배치 + 실행 순서

Algorithm:
1. ICE 스코어 내림차순 정렬
2. Layer 1 배치 (즉시 시작):
   - ICE ≥ 6.0 AND Ease ≥ 7 AND SWIFT ≥ A
   - 최대 2~3개 프로젝트
3. Layer 2 배치 (1~3개월 후):
   - ICE ≥ 4.0 AND Impact ≥ 6
   - Layer 1 프로젝트와 시너지가 있는 것 우선
4. Layer 3 배치 (3~6개월 후):
   - 나머지 중 SWIFT D등급 제외
5. Layer 1 내 실행 순서:
   - Ease 높은 것 먼저 (빠른 성공 경험)
   - 의존성 있으면 선행 프로젝트 우선
6. 병렬화 가능 그룹 식별
```

## 11. 기존 솔루션 분석 및 차별화

### 11.1 유사 도구 분석

| 도구/접근 | 특징 | 한계 |
|-----------|------|------|
| **ChatGPT/Claude 일반 대화** | 아이디어 브레인스토밍은 가능 | 구조화된 출력 없음. 실행 가능한 스펙 미생성. 반복 불가 |
| **Notion AI** | 템플릿 기반 문서 생성 | 프로젝트 분해/우선순위 로직 없음 |
| **Linear/Jira** | 프로젝트 관리 도구 | "아이디어 → 프로젝트" 변환 기능 없음 |
| **ProductBoard** | 제품 관리 + 우선순위 | B2B 엔터프라이즈 대상. 솔로 개발자에게 과다 |
| **Manual PM 작업** | 경험 많은 PM의 수동 분석 | 시간 소요 큼. 일관성 부족. 편향 존재 |

### 11.2 PM Agent의 차별점


## 12. HITL 연동

### 12.1 PM Agent의 위험 등급

| 항목 | 등급 | 근거 |
|------|------|------|
| **기본 위험 등급** | Low | 문서 생성만 수행, 코드 변경/배포 없음 |
| **Confidence 임계값** | 0.8 | 포트폴리오 전략 결정은 인간 검토 필요 |
| **자동 실행 허용** | 분석 및 스펙 생성까지 | scaffolding 호출은 인간 승인 필요 |

### 12.2 승인 필요 시점

```
자동 실행 가능:
├── 아이디어 문서 읽기 및 파싱
├── SWIFT 검증 및 ICE 스코어 산출
├── 프로젝트 스펙 문서 생성
└── plans/ 디렉토리에 파일 저장

인간 승인 필요:
├── 프로젝트 실행 순서 최종 확정
├── scaffolding 에이전트 호출
├── "폐기" 판단 (SWIFT D등급 프로젝트)
└── 예산/비용 관련 결정
```
